现代化商品与支付管理平台

一套集成了多级权限管理、自动化支付回调、商品管理与提现对账的完整商业解决方案。专为团队协作与精细化运营打造。

核心功能模块

商品管理与展示

支持富文本商品详情、多图上传、价格配置及库存管理。自动生成带支付二维码的商品分享海报，一键分发。

全自动订单对账

无缝对接第三方支付接口，实时监听支付回调 webhook。自动修改订单状态、记录支付流水，免除人工核对烦恼。

严密的权限树

内置「首席工程师-经理-主管-员工」四级权限体系。支持组织架构树状图展示，严格控制数据隔离与收益查看权限。

收益与提现管理

员工独立计算分成收益，支持发起提现申请。管理层可视化审批提现单，系统自动汇总每日/历史资金流水。

底层数据库管控

为首席工程师提供专属的「上帝视角」。网页端直连 PostgreSQL，支持查看表结构及运行自定义 SQL（增删改查）。

动态系统配置

系统字典、支付网关密钥、运行参数等支持可视化配置。修改即生效，无需重启服务器，满足多变的业务需求。

系统职责与权限体系

系统采用基于角色的访问控制（RBAC）结合数据隔离策略。不同层级的用户拥有不同的页面访问权和数据操作权。

最高权限 / 上帝视角

首席工程师 (Chief Engineer)

系统的终极管理者，拥有全部开发与运维级别的控制权限，无视所有数据隔离限制。

  • 查看全平台所有数据（包含所有商户、员工、订单）
  • 员工管理：全量增删改查所有账号（树状图/列表切换）
  • 数据库管理：直接执行 SQL 语句进行底层数据维护
  • 系统管理：配置全局环境变量、字典与核心密钥
  • 支付配置：设置系统级收款通道参数

经理 (Manager / Admin)

商户级的最高管理者，负责统筹整个商户团队的运营与资金调度。

  • 管理直属主管与员工的账号，设置提成比例
  • 查看本团队全部订单、收益及销售统计
  • 审批下属员工的提现申请
  • 配置本商户的支付网关参数

主管 (Supervisor)

团队中层管理，带领小组员工进行销售工作。

  • 创建并管理直属的下级员工账号
  • 查看自己及直属下级的订单与业绩
  • 管理自己的商品库与提现申请
  • 无法查看其他主管或上级经理的数据

员工 (Employee)

基础业务执行者，负责商品推广与客户转化。

  • 创建与管理自己的商品
  • 查看自己的专属推广订单与佣金明细
  • 发起个人的收益提现申请
  • 无任何人员管理或系统配置权限

三级收益分润机制

系统内置了极具灵活性的“自顶向下、层层抽成”的分润模型。每一笔成功支付的订单，系统都会自动沿着汇报链路（员工 ➔ 主管 截取 ➔ 经理 截取）进行 100% 的资金分配。

分配算法详解

在【员工管理】页面中，您可以为每个下属设置 收益比例 (Earnings Rate)，比如设为 50%。这个比例的含义是：“该节点（下属）能够从他拿到的资金池中，分走多少钱”，而剩下的钱则截留归他的直接上级所有。

案例演示：一笔 100 元订单的分成过程

假设架构链条为：经理 (Manager) ➔ 主管 (Supervisor) ➔ 员工 (Employee)，员工卖出了一单 100 元的商品。

  • 1
    经理给主管设置了 80% 的比例：

    100元的总池子传到主管层。经理截留 100 * (1 - 80%) = 20元。剩下的 80元 池子继续往下传给主管。

  • 2
    主管给员工设置了 50% 的比例：

    80元的池子传到员工层。主管截留 80 * (1 - 50%) = 40元。剩下的 40元 池子继续往下传。

  • 3
    最终结算：

    员工是链路的终点，拿走最后剩下的 40元。

    此单最终收益：经理 20元 | 主管 40元 | 员工 40元 （合计刚好 100 元，100% 分配完毕）

异常情况与兜底规则（未设置比例会怎样？）

情况 A：全员未设置比例 (默认为 0%)

如果下级比例都是 0%，意味着资金池“一毛不拔”往下传，下级全拿 0 元。由于最终没有被下级分走，100元将全部归属于最顶层的「经理 (Manager)」。无论如何都不会出现资金去向不明的问题。

情况 B：越级抽成 (某一层为 0%)

如果经理给主管设了 0%（主管不拿钱），但主管给员工设了 50%。
此时 100元 穿透主管到达员工层。经理拿走 100 * (1 - 50%) = 50元，员工拿走 50元，主管 0元。

系统使用手册

1 环境与部署

本项目已集成 GitHub Actions 自动化流水线。在完成开发并推送至 `main` 分支后，服务器会自动完成部署：

git commit -m "update"
git push origin main
# 自动触发云服务器拉取代码 -> npm build -> pm2 restart

手动运维：如果需要手动干预，请登录服务器并执行 pm2 logs payforme 查看实时运行日志。

2 初始化设置 (首席工程师必看)

作为首席工程师，接手系统后建议首先进行以下操作：

  • 进入 系统管理 页面，配置全局需要的系统变量（如 SYSTEM_TITLE 等）。
  • 进入 数据库管理 页面，检查表状态。可执行 SELECT * FROM users; 核对核心账号。
  • 进入 员工管理，清理多余的测试账号。您可使用“列表视图”进行直观的批量管理，点击垃圾桶即可安全删除。

3 日常运营流程

  1. 建品：在「商品管理」新建商品，上传封面，设置价格。
  2. 分发：点击商品生成推广海报，分发给客户。
  3. 支付：客户扫码支付后，Webhook 自动回调更新「订单管理」状态。
  4. 分润：系统按预设比例计算佣金，计入员工「我的钱包」。
  5. 提现：员工发起提现，经理/财务进行审批打款。

4 异常排查指南

  • 界面显示异常/未更新： 请使用 Ctrl + F5 强制清空浏览器缓存加载最新 JS 文件。
  • 接口报 500 错误： 请首席工程师登录服务器查看 PM2 日志，或在「数据库管理」模块使用 SQL 执行器排查脏数据。
  • 订单状态未变更： 检查支付服务商的 Webhook 回调地址配置是否正确，且网络是否畅通。
