# PHPWC 支付通道预览测试 Spec

## Why
在前面的开发中，我们已经完成了 PHPWC 支付通道的代码开发并推送到 `feature/add-phpwc-payment-channel` 分支。为了确保项目上线后该通道能够正常、稳定地运作，并且与其他已有功能（如订单流转、现有支付通道等）完美兼容，我们需要在这个新分支上搭建本地预览环境，并进行全链路的自动化和手动测试。

## What Changes
- 启动本地全栈开发服务器（Frontend + Backend）。
- 编写一个模拟客户端发起 PHPWC 支付请求的测试脚本，验证测试 PID(199) 的 0.1 元特判逻辑。
- 编写一个模拟 PHPWC 官方发起异步回调（Webhook）的测试脚本，验证 MD5 验签及订单状态更新机制。
- 暴露本地域名预览链接供用户手动点击测试体验。

## Impact
- Affected specs: 开发环境启动流程、支付功能自测流程。
- Affected code: 项目本身代码不被修改，仅新增测试脚本文件用于验证，并在验证后清理。

## ADDED Requirements
### Requirement: 预览环境搭建与支付全链路验证
系统 SHALL 能够在本地成功启动，支持前台页面访问与后台接口调用，并且允许自动化脚本或用户通过预览链接完成“创建订单 -> 验证签名 -> 模拟支付回调 -> 订单成功”的完整生命周期。

#### Scenario: 发起 PHPWC 测试订单
- **WHEN** 自动化测试脚本或用户通过预览链接下单并选择 `phpwc` 支付
- **THEN** 接口返回的支付 URL 必须包含正确的 MD5 签名，且在测试环境（PID:199）下支付金额（`money`）必须被强制设为 `0.1`。

#### Scenario: 模拟异步回调成功
- **WHEN** 测试脚本向系统的 webhook `/api/orders/phpwc/callback` 发送构造好的、签名正确的伪造支付成功通知
- **THEN** 系统能够正确通过验签，拦截重放攻击，并将对应订单的状态从 `pending` 变更为 `paid`。
