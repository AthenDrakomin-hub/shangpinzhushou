# 支付通道配置维护指南

本指南详细记录了商户系统中“三方支付动态通道管理”的高级功能原理及维护方式。通过阅读本文档，您可以清楚地了解如何使用后台管理现有的通道、系统内部的价格匹配逻辑，以及未来如何通过修改代码接入全新的三方支付网关。

---

## 1. 动态通道管理概述

为了应对多变的三方支付环境（如通道维护、限额调整、新增业务线等），系统在管理后台实现了完全解耦的**动态通道管理系统**。

**核心功能**：
- **无需修改代码即可上线新通道**：只要属于现有的底层网关（如 `SuperPay` 或 `九久支付`），仅需在后台【设置】页面配置“集合编码”、“显示名称”和“限额区间”即可立刻生效。
- **全链路价格自动校验**：
  1. **商品创建/编辑时拦截**：输入的金额若不在任何通道的 `minAmount` 和 `maxAmount` 之间，则会标红警告并禁止创建商品。
  2. **商品级通道开启**：系统会动态列出符合价格区间的支付通道，商户需自行勾选该商品允许使用的通道。
  3. **H5 收银台智能渲染**：买家在购买页只能看到该商品所选且符合其价格的通道，彻底防止通道路由错乱（如低价唤起 USDT）。

## 2. 后台配置操作指南

### 2.1 进入通道管理
1. 使用具有 `manager`（经理）或 `chief_engineer`（首席工程师）权限的账号登录系统。
2. 在侧边栏导航点击 **“设置 (Settings)”**。
3. 找到 **“动态通道管理”** 卡片，点击“管理支付通道”按钮。

### 2.2 通道字段说明
在弹出的通道管理表格中，每个通道包含以下关键信息：

| 字段名 | 说明 | 示例 |
|---|---|---|
| **通道名称** | 展示给员工和买家看的友好名称。 | `支付宝扫码 (小混)` |
| **网关类型** | 决定这笔订单在底层走哪套 API 逻辑。目前系统内置了 `SuperPay` (支付宝/USDT) 和 `九久支付` (微信)。 | `superpay` |
| **通道编码** | 三方网关下发的集合编码/通道标识，这是拉起对应支付方式的核心参数。 | `824`、`6007`、`1` |
| **最小金额** | 该通道允许的最低交易额（包含）。 | `100.00` |
| **最大金额** | 该通道允许的最高交易额（包含）。 | `20000.00` |

### 2.3 典型配置示例

- **SuperPay 支付宝正式通道**
  - 网关类型：`SuperPay`
  - 通道编码：`824`
  - 限额：`100` - `20000`
- **SuperPay 支付宝测试通道**
  - 网关类型：`SuperPay`
  - 通道编码：`1`
  - 限额：`1` - `100`
- **九久微信小额原生通道**
  - 网关类型：`九久支付`
  - 通道编码：`6007`
  - 限额：`1` - `50`

*配置保存后，将立即对所有新创建的商品以及正在访问的买家生效。*

## 3. 技术架构与数据流转

本系统的支付通道配置存储在数据库的 `system_configs` 表中，以 `JSON` 格式统一存储，方便随时扩充字段。

### 数据流向图
1. **后台配置**：管理员在 `SettingsPage.tsx` 增删改查通道数据，提交至 `PUT /api/admin/payment-channels`。
2. **商品校验**：`ProductCreatePage.tsx` 请求 `GET /api/admin/payment-channels`，比对输入的价格，勾选后的结果拼接成字符串保存在商品的 `supported_pay_methods` 字段中（例如：`ch_123,ch_456`）。
3. **前端收银台**：买家访问 `H5ProductPage.tsx`（无需登录），前端请求 `GET /api/payment-channels`，与商品的 `supported_pay_methods` 取交集，并根据当前售价再次过滤，只显示合法的支付按钮。
4. **后端下单网关路由**：
   - 收到 `POST /api/orders`，提取 `payType` (此时为前端传来的通道唯一 `id`)。
   - 后端查询全局通道配置，校验 `channel.minAmount <= price <= channel.maxAmount`。
   - 判断 `channel.gateway`：
     - 若为 `superpay`，则调用 `src/services/superPay.ts`，并将 `channel.channelCode` 传入。
     - 若为 `jiujiu`，则调用 `src/services/wechatPay.ts`，并将 `channel.channelCode` 传入。

## 4. 开发者：如何接入全新的三方网关

如果未来公司需要接入除了 `SuperPay` 和 `九久支付` 之外的第三家支付平台（假设叫 `StarPay`），请按照以下步骤进行二次开发：

### 第一步：实现底层服务 API
在 `src/services/` 目录下新建 `starPay.ts`。参考现有的 `superPay.ts`，实现签名算法和下单网络请求。
```typescript
// src/services/starPay.ts
export async function createStarPayOrder(params: any, baseUrl: string) {
  // 1. 签名逻辑
  // 2. 发送 POST 请求到 StarPay
  // 3. 返回统一个格式: { success: true, payUrl: '...', formHtml: '...' }
}
```

### 第二步：注册新的网关类型 (Frontend)
在前端的设置页面中，给“网关类型”下拉框增加一个新的选项。
打开 `src/pages/SettingsPage.tsx`，找到 `PaymentChannelsModal` 组件中的下拉菜单代码：
```tsx
<select value={editingChannel.gateway} ...>
  <option value="superpay">SuperPay</option>
  <option value="jiujiu">九久支付</option>
  <option value="starpay">StarPay (新网关)</option> {/* 新增这行 */}
</select>
```

### 第三步：配置后端路由网关 (Backend)
打开核心下单文件 `server.ts`，定位到 `app.post('/api/orders', ...)` 路由：

1. **导入新网关服务**
```typescript
const { createStarPayOrder } = await import('./src/services/starPay.js');
```

2. **在路由中添加 `else if` 分支**
```typescript
// server.ts -> 约 1250 行左右
if (channel.gateway === 'superpay') {
  // ... 现有的 superpay 逻辑
} else if (channel.gateway === 'jiujiu') {
  // ... 现有的 jiujiu 逻辑
} else if (channel.gateway === 'starpay') {
  // ========== 使用 StarPay ==========
  console.log('[订单创建] 使用 StarPay, 通道编码:', channel.channelCode);
  
  const payResult = await createStarPayOrder({
    orderSn: orderId,
    amount: product.price.toString(),
    channelCode: channel.channelCode,
    notifyUrl: `${projectDomain}/api/orders/starpay/callback`,
    returnUrl: `${projectDomain}/payment/result?orderId=${orderId}`
  }, config.starPayBaseUrl);

  if (!payResult.success) {
    return res.status(400).json({ error: payResult.error });
  }

  payUrl = payResult.payUrl || '';
  formHtml = payResult.formHtml || '';
}
```

### 第四步：编写回调接口 (Webhook)
在 `server.ts` 中新增对应网关的异步通知路由（如 `app.post('/api/orders/starpay/callback', ...)`），负责校验签名并更新订单状态为 `paid`。

完成以上四步后，运营人员即可在后台的“动态通道管理”中自由添加属于 `StarPay` 网关的任意渠道集合，系统将自动接管其生命周期和限额校验。
