## 自动化部署 (CI/CD)

项目已配置 GitHub Actions 自动化部署。

**触发条件**：
向 GitHub 仓库的 `main` 分支推送代码，或者在 Actions 面板手动触发。

**环境依赖 (GitHub Secrets)**：
必须在仓库的 Settings -> Secrets and variables -> Actions 中配置以下 Secret：
- `VULTR_HOST`: `167.179.87.106`
- `VULTR_USERNAME`: `root`
- `VULTR_PASSWORD`: 您的服务器 SSH 密码

**自动化流程**：
1. 通过 SSH 密码连接服务器。
2. 进入项目目录并执行 `git pull origin main`。
3. 执行 `pnpm install` 与 `pnpm build`。
4. 使用 `pm2` 重启后端服务。

## Nginx 配置

配置文件：`/etc/nginx/sites-available/goodspage`

### 配置要点

1. **HTTP 自动跳转 HTTPS**：所有 HTTP 请求自动重定向
2. **反向代理**：所有请求代理到 `localhost:5000`
3. **WebSocket 支持**：已配置 `Upgrade` 头
4. **安全头**：X-Frame-Options, X-Content-Type-Options, X-XSS-Protection

### 配置内容

```nginx
# HTTP 重定向到 HTTPS
server {
    listen 80;
    server_name goodspage.cn www.goodspage.cn;
    
    # Let's Encrypt 验证路径
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }
    
    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS 配置
server {
    listen 443 ssl http2;
    server_name goodspage.cn www.goodspage.cn;
    
    ssl_certificate /etc/nginx/ssl/goodspage.cn.fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/goodspage.cn.key;
    
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;
    
    # 安全头
    add_header X-Frame-Options SAMEORIGIN always;
    add_header X-Content-Type-Options nosniff always;
    add_header X-XSS-Protection "1; mode=block" always;
    
    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
```

## 部署流程

### 首次部署

```bash
# 1. SSH 连接服务器
ssh root@167.179.87.106

# 2. 创建项目目录
mkdir -p /root/payforme
cd /root/payforme

# 3. 同步项目文件（从本地）
# 在本地执行：
rsync -avz --exclude 'node_modules' --exclude '.git' \
  ./ root@167.179.87.106:/root/payforme/

# 4. 安装系统依赖（canvas 需要）
apt-get install -y build-essential libcairo2-dev libpango1.0-dev libjpeg-dev libgif-dev librsvg2-dev

# 5. 安装 Node.js 依赖
pnpm install

# 6. 安装 canvas（需要本地编译）
npm install canvas --build-from-source

# 7. 配置环境变量
# 编辑 .env 或 ecosystem.config.cjs

# 8. 构建前端
pnpm build

# 9. 启动服务
pm2 start ecosystem.config.cjs

# 10. 保存 PM2 配置
pm2 save
```

### 更新部署

```bash
# 方式一：使用 rsync（推荐）
rsync -avz --exclude 'node_modules' --exclude 'uploads' --exclude '.env' \
  ./ root@167.179.87.106:/root/payforme/

# 方式二：使用 Git
cd /root/payforme && git pull

# 更新后操作
pnpm install        # 更新依赖
npm install canvas --build-from-source  # 重新编译 canvas
pnpm build          # 重新构建
pm2 restart payforme # 重启服务
```

## 故障排查

### 服务无法启动

```bash
# 1. 检查端口占用
ss -tlnp | grep 5000

# 2. 检查日志
pm2 logs payforme --lines 50

# 3. 检查数据库连接
psql -U payforme -d payforme -c "SELECT 1"

# 4. 手动启动测试
cd /root/payforme && pnpm run dev
```

### 数据库连接失败

```bash
# 1. 确认 PostgreSQL 服务运行
systemctl status postgresql

# 2. 检查数据库用户权限
psql -U postgres -c "\du"

# 3. 检查数据库是否存在
psql -U postgres -c "\l"

# 4. 重启 PostgreSQL
systemctl restart postgresql
```

### HTTPS 证书问题

```bash
# 1. 检查证书文件
ls -la /etc/nginx/ssl/

# 2. 测试证书
openssl x509 -in /etc/nginx/ssl/goodspage.cn.fullchain.pem -text -noout

# 3. 重载 Nginx
nginx -s reload
```
